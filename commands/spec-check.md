# /whale:spec-check — 요구사항 변경 절차 판단

변경 요청: **$ARGUMENTS**

## 판단 기준 (프로젝트 CLAUDE.md "요구사항 변경 시 필수 절차")

두 질문 중 **하나라도 "예"** 이면 코드 전에 spec 을 먼저 수정한다:
1. 다른 에이전트(FE↔BE) 또는 다른 BC 에 영향을 주는가?
2. 기존 문서를 읽으면 이 변경을 예측할 수 없는가?

## 변경 유형별 갱신 대상 (프로젝트 CLAUDE.md 의 매핑 표를 우선한다)

| 변경 내용 | 갱신할 문서(일반형) |
|----------|-----------|
| 신규 기능 추가 | PRD + 해당 도메인 모델 |
| 기존 기능 동작 변경 | 해당 도메인의 business-rules 문서 또는 마일스톤 문서 |
| 공통 컴포넌트화·모듈 재설계 | ARCHITECTURE + ADR (결정 이유 기록) |
| API 계약 변경 | 해당 기술 스펙 문서 |
| 마일스톤 범위 영향 | `config.json.scopeDir` — **사용자 확인 필수** |

## 출력

위 요청에 대해 판단해 보고한다:
1. **버그픽스/리팩토링인가, 기능 추가/변경/확장인가?** (전자면 spec 갱신 불필요 — 그대로 종료)
2. 기능 변경이면 위 두 질문의 답(예/아니오)과 근거.
3. spec-first 필요 시: **갱신할 문서 목록** + 각 문서에 무엇을 추가/수정할지 요지.
4. 마일스톤 범위 영향이 있으면 사용자 확인 필요를 명시.

판단·산출만 한다 — 실제 문서 수정이나 코드 작성은 하지 않는다.
