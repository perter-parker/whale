🐳 Whale 사용 매뉴얼

역할 기반 AI 에이전트 팀을 슬래시 명령어로 구동하는 워크플로 프레임워크. 작업 한 건의 생애주기(조사→계획→승인→구현→리뷰→QA→요약)를 품질 게이트와 함께 자동화하는 Claude Code 플러그인입니다.

생애주기 품질 루프run-id 승인 게이트무인 E2E 트리아지 SAST/SCA 보안강제 Hooks계약 우선교차-run 메모리 병렬 worktree슬래시 없이 자동 라우팅

1 Whale이란

Whale은 개발 작업을 역할이 나뉜 AI 에이전트들과 그걸 엮는 오케스트레이션 명령어로 수행합니다. 혼자 개발해도 "조사원·기획자·개발자·리뷰어·QA·보안·기록자"로 이뤄진 나만의 팀을 갖는 셈입니다.

설계 철학

🔒 컨텍스트 격리

각 단계는 독립 실행. 리뷰어·QA는 구현 맥락에 물들지 않고 산출물만 보고 객관 판정.

✋ 강제 승인 게이트

계획은 사람이 APPROVED: <run-id>를 명시해야 구현 진입. "ok"·"진행해"는 불허.

🔁 명시적 피드백 루프

리뷰/QA가 "재구현 필요 YES/NO" 강제 출력. YES면 지적 전문가만 되돌림(최대 1회).

📝 기록 강제

모든 단계 산출물을 파일로 남기고, 결정·교훈을 run 간 누적.

🧪 검증이 병목

리뷰·보안은 "권고"가 아니라 Hooks로 "무조건 실행"되는 게이트.

🧩 데이터 주도 이식성

역할·경로·순서를 하드코딩하지 않고 config.json에서 읽음. 어느 프로젝트든 재사용.

범용 기본 + 선택적 도메인 프로필. 기본은 범용입니다 — whale 은 특정 아키텍처나 표준을 강제하지 않고, 각 에이전트가 프로젝트 CLAUDE.md 를 읽어 따릅니다. 공공기관 프로젝트라면 config.profile.domain=public-sector-kr 로 켜서 dba 에 내장된 행정안전부 DB 표준화 지침(표준도메인·감사컬럼·논리삭제·PII 암호화)을, multitenant=true 로 site_id 격리를 적용합니다. 아키텍처(레이어드/DDD/피처슬라이스)는 전적으로 프로젝트가 결정합니다.

2 설치 & 초기화

플러그인 설치

# 1) 마켓플레이스 등록 (로컬 경로 또는 GitHub)
/plugin marketplace add /path/to/whale-plugin
# 또는 /plugin marketplace add <github-user>/whale-plugin

# 2) 플러그인 설치
/plugin install whale

프로젝트 초기화 (프로젝트마다 1회)

명령어무엇을 스캐폴드하나필수?
/whale:init .claude/whale/에 config.json·state.md·runs/·memory/ + (부재 시) 프로젝트 CLAUDE.md 필수
/whale:hooks-init 프로젝트 .claude/settings.json에 lint/test 게이트 + 위험명령 차단 훅 (스택 감지·병합) 권장
/whale:e2e-init Playwright 설정·Helper 스켈레톤·fixture·AI 가이드·CI 워크플로 E2E 쓸 때
안전 원칙. 초기화 명령들은 기존 파일을 절대 덮어쓰지 않습니다. 이미 있으면 병합안을 제시하거나 건너뜁니다. settings.json은 사람이 승인해 병합합니다.

초기화 후 채울 것

  • CLAUDE.md — 기술 스택·패키지 매니저·동결 대상·§3 계약 SSOT·§7 FEATURE.md 관례.
  • config.json — paths(화면정의서·PRD 경로), 필요 시 e2e·security 블록.
  • 각 핵심 모듈에 FEATURE.md(owns/does-not-own/진입점/이벤트/의존성) 배치.

슬래시 없이 자동 사용 (기본 동작)

/whale:* 를 매번 치지 않아도 개발 요청이 자동으로 whale 흐름으로 진행됩니다. 3개 레버가 기본 동작을 whale로 기울입니다 — 승인 게이트는 그대로 유지됩니다.

레버동작활성화
Output Style (내장) output-styles/whale.md — "OAuth 로그인 만들어줘"만 해도 researcher→planner→⛔승인→구현→리뷰→qa→요약으로 흐름 force-for-plugin으로 플러그인 켜면 자동. /output-style로 확인·전환
CLAUDE.md 라우팅 프로젝트 CLAUDE.md §0 + 전역 ~/.claude/CLAUDE.md가 매 세션 라우팅 규칙 로드 /whale:init이 프로젝트에 심음 · 전역은 스니펫 블록 추가
에이전트 트리거 "이 코드 리뷰해줘"→reviewer, "테이블 추가"→dba 처럼 단일 작업은 바로 해당 전문가로 자동 위임 각 에이전트 description에 "…할 때 사용" 키워드 내장
알아둘 점. Claude Code에 "강제" 자동 위임은 없습니다 — 모델이 매번 판단합니다. 위 레버는 기본값을 강하게 기울일 뿐이고, 100% 결정적으로 가려면 여전히 /whale:start·/whale:next가 확실합니다. Output Style·description 변경은 Claude Code 재시작 또는 /reload 후 반영됩니다. 사소한 작업(오타·1줄)은 생애주기 없이 바로 처리됩니다.

3 핵심 개념

생애주기 7단계 (Mode A)

research→ plan→ approve ✋→ implement→ review→ qa→ summarize

implement는 계획에 등장하는 도메인 전문가(dba→be→fe)만 순차 호출합니다. E2E·보안은 별도 단계가 아니라 qa가 필요 시 부릅니다.

run-id

/whale:start 시 발급되는 작업 식별자(예: 20260716-oauth-login-01). 모든 산출물은 .claude/whale/runs/<run-id>/에 파일로 남고, 승인·상태 추적의 키가 됩니다.

3종 게이트

게이트적용 단계통과 조건
artifactresearch·plan·implement·summarize산출물 파일이 실제 내용으로 존재
approvalapprove사람이 APPROVED: <run-id> 명시 (느슨한 승인 불허)
verdictreview·qa재구현 필요: YES/NO 판정. YES면 피드백 루프

피드백 루프

review 또는 qa가 YES를 내면 → 지적된 전문가만 implement로 되돌려 지적 범위만 수정(전체 재구현 아님). 최대 1회, 초과 시 리더에게 에스컬레이션(무한 루프 방지).

qa-conditional 이란? E2E(e2e-tester)와 보안(security-reviewer)은 독립 lifecycle 단계가 아닙니다. qa 단계가 critical path/보안 민감 변경을 감지하면 내부에서 호출하고, 그 결과(APP_BUG·Critical/High)를 자신의 판정에 흡수합니다. 그래서 게이트는 여전히 qa.md 하나입니다.

4 생애주기 워크플로 (Mode A)

신규 기능 개발의 표준 흐름입니다. 각 단계는 /whale:next로 넘어갑니다.

단계별 진행

#명령일어나는 일산출물
1/whale:start <작업설명>run-id 발급 → researcher 조사 시작research.md
2/whale:nextplanner가 7섹션 계획서 작성 + 승인 요청plan.md
3/whale:approve <run-id>사람이 계획 승인 (또는 --reject로 재계획)—
4/whale:next ×N계획에 배정된 dba→be→fe를 한 번에 하나씩 TDD 구현코드·테스트
5/whale:nextreviewer가 아키텍처·품질·부분수정 검토 (YES면 4로 복귀)review.md
6/whale:nextqa가 AC·회귀·여정 검증 (+조건부 E2E·보안)qa.md
7/whale:nextsummarizer가 작업 결과지(PR 디스크립션) 작성 → 리더 검수summary.md

예시 세션

/whale:start OAuth 소셜 로그인 추가
  → run-id: 20260716-oauth-login-01 발급, research.md 생성

/whale:next   # 계획 수립
  → plan.md (P1~P7) + "승인하려면 /whale:approve 20260716-oauth-login-01"

/whale:approve 20260716-oauth-login-01
  → 승인됨. 이제 implement 진입

/whale:next   # dba 스키마 (계획에 dba가 있으면)
/whale:next   # be API (TDD)
/whale:next   # fe 화면 (TDD)
/whale:next   # reviewer → 재구현 필요: NO
/whale:next   # qa → Go, (로그인=critical → e2e·보안 자동 호출)
/whale:next   # summarizer → summary.md, 리더 최종 검수

7섹션 계획서 (plan.md)

planner가 만드는 개발 정본. reviewer·next가 파싱합니다.

섹션내용
P1 목표·범위AC(수용 기준)·BR-ID(비즈니스 규칙)·상위 근거
P2 영향 범위 + 계약 우선변경 대상 파일 + 계약 SSOT 표(경계별 원본→파생물→codegen)
P3 구현 단계필요한 도메인만 순서·의존
P4 전문가 배정dba/be/fe 담당 여부 → implement dispatch 필터
P5 테스트 전략TDD Red 테스트 + E2E critical path
P6 리스크·제약동결·PII·site_id·BC 경계·마이그레이션 버전
P7 완료 정의DoD 체크리스트

진행 상황 확인

/whale:status — run-id·현재 phase·승인 상태·재시도 횟수·다음 액션을 표시합니다.

5 E2E 무인 테스트

야간·무인 상황에서 AI에게 테스트를 맡기기 위한 하네스입니다. Playwright 영속 스펙으로 critical path를 실행하고, 실패를 4분류로 트리아지합니다.

준비 (최초 1회)

/whale:e2e-init
  # 스택 감지(pnpm/npm) → 생성:
  #   playwright.config.ts
  #   tests/helpers/{form,dialog,select,table,navigation,toast}.helper.ts
  #   tests/fixtures/index.ts  (Helper 자동 주입)
  #   tests/e2e/example.spec.ts
  #   tests/README.e2e.md      (= AI 가이드)
  #   .github/workflows/e2e.yml (라벨 기반 CI)

# 의존성 설치는 안내만 (사용자 확인):
pnpm add -D @playwright/test && npx playwright install

실행

qa 단계가 critical path를 감지하면 자동 호출하거나, 직접 /whale:e2e <시나리오>로 실행합니다.

Self-healing 4분류 핵심

분류의미AI 행동
UI_CHANGE의도적 UI 변경으로 셀렉터 어긋남셀렉터 자가수정 (최대 maxHealRetries회)
TEST_BUG테스트 코드 결함(스코프·대기)테스트 자가수정
APP_BUG앱의 실제 결함고치지 않고 리포트 → qa Blocker → 재구현
ENV_ISSUE서버 미기동·네트워크고치지 않고 리포트
무인 안전. e2e-tester는 앱 코드·마이그레이션·서버를 절대 변경하지 않습니다(E2E 스펙·Helper·리포트만). 파괴적/비가역 시나리오(실데이터 삭제·결제·메일)는 실행하지 않고 스킵합니다.

셀렉터 우선순위 (내장 규칙)

  1. getByRole('button', {name:'저장'}) — ARIA role (1순위)
  2. getByLabel('이름') — ARIA 속성 (2순위)
  3. [data-slot="dialog-title"] — 디자인시스템 (3순위)

금지: getByText(/A|B|C/) 다중매칭(strict mode 위반), CSS/XPath 절대경로.

결과: 트리아지 리포트 e2e.md

진짜 앱버그(APP_BUG)를 크리티컬리티순 최상단에 두어 아침에 5초 안에 판정할 수 있게 합니다.

6 강제 게이트 & 보안

Hooks — "권고"를 "강제"로

Whale의 리뷰·QA는 에이전트 "판단"이지만, Hooks는 "무조건 실행"입니다.

/whale:hooks-init
  # .claude/settings.json 에 병합(덮어쓰기 금지):
  #  PreToolUse(Bash)      → 위험명령 차단 (exit 2)
  #  PostToolUse(Edit|Write) → lint·typecheck 게이트 (기본 lint-only)

기본 차단 명령: rm -rf /~., 보호브랜치 git push --force, prod 마이그레이션, DROP/TRUNCATE, curl|sh, terraform destroy, kubectl delete namespace 등.

게이트 모드: lint-only(기본, 무인 빠름) · full(test 포함) · changed-only · off. 전체 test는 QA 단계에 위임합니다.

보안 검토 (security-reviewer)

qa가 보안 민감 변경(인증·PII·site_id·의존성·결제)을 감지하면 /whale:security로 호출합니다. 산출물 security.md.

축점검
SAST인젝션·XSS·역직렬화·인증/인가 우회(JWT·RBAC)·시크릿 하드코딩·입력검증·PII 노출·site_id 테넌시
SCA신규 패키지 실존 확인(환각/슬롭스쿼팅 방어)·취약버전(CVE)·라이선스·버전 고정
판정의미다음
FAILCritical/High 존재 (머지 차단)qa Blocker → 재구현 YES
HOLD인증·암호화·입력검증·결제 변경 (무결점이어도)리더 보안 승인 대기 — 자동 통과 금지
PASSCritical/High 없음 + 사람게이트 무저촉qa 종합 판정 진행
사람 필수 게이트. 인증·암호화·입력검증·결제 관련 변경은 취약점이 0건이어도 무인 자동 통과되지 않고 HOLD됩니다. 공공기관 감사 추적성(누가 언제 보안 승인했는가)을 위한 안전장치입니다.

계약 우선 (Contract-first)

레이어 경계마다 "진실은 여기 하나"(SSOT)를 못박아 FE/BE 드리프트를 차단합니다. planner가 config.contracts.ssot/CLAUDE.md §3에 지정하고, be/fe가 준수합니다.

경계SSOT(원본)파생물
APIOpenAPI / BE DTOFE 타입·클라이언트
DB마이그레이션 DDL엔티티·타입
도메인스키마/zodFE 모델

규칙: SSOT를 바꾸면 같은 커밋에서 파생물을 갱신(가능하면 codegen). 파생물만 손대는 것은 드리프트=금지.

교차-run 메모리

.claude/whale/memory/에 run 간 학습을 누적합니다.

  • PROGRESS.md — 진행 로그
  • DECISIONS.md — 결정 + 이유 (be/fe가 append)
  • LessonsLearned.md — 반복 실패 패턴 (reviewer가 append, summarizer가 정리, 다음 researcher가 승계)

7 이슈 해결(Mode B) & 부채 상환

Mode B — 이슈 해결

/whale:fix <이슈 설명> — 고정 순서 없이 이슈 성격에 맞는 역할(들)을 피어로 투입합니다. run-id 생애주기와 독립입니다.

  • API 불일치 → be ↔ fe
  • 테스트/품질 의문 → /whale:review·/whale:qa
  • 도메인 규칙 의문 → /whale:domain
  • 기능 변경이면 먼저 /whale:spec-check로 spec-first 절차 판단

부채 상환 (refactor)

/whale:refactor [대상] — 생애주기와 독립. 복잡도·중복·대형 모듈을 정리하고 FEATURE.md를 생성합니다.

동작 불변 강제. 리팩터 전후 동일한 테스트가 동일하게 통과해야 합니다. 기준선이 없으면 특성화 테스트를 먼저 추가합니다. 기능 추가·동작 변경은 금지(섞이면 /whale:start·/whale:fix로). 위험 큰 분해는 제안만 하고 리더 승인 대기.

8 병렬 worktree 오케스트레이션

터미널 여러 개로 병렬 작업할 때 하나의 워킹 디렉토리를 공유해 소스가 섞이는 문제를, 작업별 git worktree(폴더·브랜치 물리 격리)로 해결합니다. 각 worktree 폴더에서 터미널을 열면 그 안의 whale 팀은 자기 폴더·자기 브랜치만 봅니다.

생성 & 사용

/whale:worktree create main login payment search notice
  # ../<repo>-login, -payment, -search, -notice 폴더를
  # 각각 feature/login … 브랜치로 생성 + 의존성 설치 + .env.worktree(PORT)

# 그다음 각 폴더에서 터미널을 하나씩:
cd ../<repo>-login   && claude     # feature/login
cd ../<repo>-payment && claude     # feature/payment

관리

명령동작
/whale:worktree create <base> <name...>폴더·브랜치 생성 + 의존성 설치 + PORT 분리
/whale:worktree list현재 worktree 목록 (+PORT)
/whale:worktree remove <name> [--force]정리 (브랜치는 유지)
설계. worktree 세팅은 결정적 오케스트레이션이라 command+script 층이 담당하고, 개발 지능은 각 worktree 안의 taskforce 에이전트가 담당합니다(생애주기·run-id와 독립). whale을 유저 레벨(~/.claude/)에 설치하면 갈라진 폴더마다 자동으로 붙습니다.
worktree가 격리하지 못하는 것.
  • 의존성/빌드는 폴더마다 따로입니다(pnpm 등 전역 캐시면 부담↓).
  • 같은 브랜치를 두 worktree에서 동시 체크아웃 불가(git이 막습니다).
  • 로컬 DB/포트는 공유되면 충돌 → .env.worktree의 PORT=3001, 3002…를 참고해 dev 서버를 다르게 띄우세요.
  • 머지 차단 게이트(테스트·보안 스캔)는 이 층이 아니라 CI(.gitlab-ci.yml 등)에서 강제하세요.
  • 새 worktree에 .claude/whale/config.json이 없으면(git 미커밋) 그 폴더에서 /whale:init을 먼저 실행하세요.

13 FAQ · 트러블슈팅

Q. /whale:next가 "승인 대기"로 멈춰요.

approve 게이트입니다. /whale:approve <run-id>를 실행하세요. "ok"·"진행해" 같은 표현은 인정되지 않습니다.

Q. 재구현이 반복돼서 멈췄어요.

피드백 루프는 최대 1회입니다. 초과하면 자동 진행이 멈추고 리더 개입을 요청합니다. 직접 수정하거나 /whale:approve --reject로 재계획하세요.

Q. qa가 "리더 보안 승인 대기(HOLD)"라고 해요.

인증·암호화·입력검증·결제 변경은 무결점이어도 자동 통과되지 않습니다. security.md를 확인하고 사람이 명시 승인해야 진행됩니다.

Q. E2E/보안을 안 쓰고 싶어요.

config.json에서 e2e/security 블록을 비우거나 제거하면 됩니다. qa는 커버리지 공백을 표기하고 그냥 진행합니다.

Q. 공공기관 프로젝트가 아닌데 행안부 표준·site_id가 강제되나요?

아니요. 기본은 profile.domain=general·multitenant=false라 행안부 표준·PII 암호화·site_id는 전혀 적용되지 않습니다(dba의 부록 A는 무시됨). 공공기관 프로젝트일 때만 config.profile에서 켜세요. 아키텍처도 whale이 강제하지 않고 프로젝트 CLAUDE.md가 결정합니다.

Q. worktree를 슬래시 없이 자동으로 만들 수 있나요?

네. 여러 기능/도메인을 병렬로 진행하려는 신호가 보이면 Claude가 worktree 세팅을 제안·실행합니다(폴더·의존성 설치 부작용이 있어 확인 후 실행). 슬래시 커맨드는 수동 진입점일 뿐이고, 내부적으로 whale-worktree.sh를 실행합니다.

Q. Hooks가 정상 명령까지 막아요.

가드는 파괴적 패턴만 exit 2로 막습니다. 의도한 위험 작업이면 사람이 터미널에서 직접 실행하세요(! 명령). 오탐이면 config.hooks.guard.blockPatterns를 조정하세요.

Q. refactor가 기능을 못 고친다는데요?

맞습니다. refactor는 동작 불변만 합니다. 기능 변경은 /whale:start(신규) 또는 /whale:fix(이슈)로 진행하세요.

Q. 매번 /whale:*를 쳐야 하나요?

아니요. 내장 Output Style(force-for-plugin)과 CLAUDE.md 라우팅이 개발 요청을 자동으로 whale 흐름으로 보냅니다("2. 슬래시 없이 자동 사용" 참고). 단 승인 게이트는 유지되므로 계획 후 승인은 직접 해주세요. Output Style 변경은 /reload 후 반영됩니다.

Q. 자동으로 안 도는데요?

Claude Code에 강제 자동 위임은 없어(모델이 판단) 가끔 안 탈 수 있습니다. /output-style로 whale 스타일이 켜졌는지 확인하고, 확실히 태우려면 /whale:start <작업>으로 시작하세요.

Q. 다른 프로젝트에서도 쓰려면?

플러그인은 그대로 두고 각 프로젝트에서 /whale:init 한 번 실행 후 config.json·CLAUDE.md만 그 프로젝트에 맞게 채우면 됩니다. 전역 ~/.claude/CLAUDE.md의 whale 라우팅 블록은 모든 프로젝트에 적용됩니다.

Q. /whale:worktree remove가 안 돼요.

해당 worktree에 커밋 안 된 변경이 있으면 안전하게 거부됩니다. 작업을 커밋/스태시하거나, 버릴 거면 remove <name> --force를 쓰세요. 브랜치는 자동 삭제되지 않으니 머지 후 git branch -d로 지우면 됩니다.

Q. 새 worktree에서 dev 서버가 포트 충돌해요.

worktree는 폴더·브랜치만 격리합니다. 생성 시 만든 .env.worktree의 PORT(3001, 3002…)를 프로젝트 방식으로 로드해 서버를 다르게 띄우세요.
